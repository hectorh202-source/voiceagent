# Google Local Services Ads (LSA) leads

Google Local Services Ads produces two kinds of inbound inquiries — `MESSAGE` (a text inquiry through the ad) and `PHONE_CALL` (a tracked, billed phone call) — for a business running LSA campaigns. This is the third lead source feeding the [Leads inbox](leads-inbox.md), after the generic website form/chat webhook. Unlike that webhook, Google's API is **poll-based, not push** — there's no true webhook for LSA leads, so this is (once built) a periodic poller, the same shape as [the Twilio recording poller](call-dashboard.md#why-polling-not-a-webhook--a-real-incident-and-what-was-actually-tried) rather than an inbound route.

**Status: live and verified end-to-end against TitanZ's real Google Ads account (2026-07-17).** `src/googleLsa/` is fully built, wired into `src/index.ts`'s 5-minute poll loop, and has pulled real `MESSAGE`/`PHONE_CALL` leads (including a full real customer message thread and a real call recording URL) into `inbound_leads`, decrypted correctly on read.

## Sequencing — why this needed a manual setup stage first

Every other integration in this codebase (ServiceTitan, ElevenLabs, Twilio) could be verified against real data with nothing more than an already-configured account and a scratch script. Google's Local Services Ads API needed real, manual setup on Google's side first — a Developer Token (needs Google's approval), an OAuth 2.0 Client ID/Secret, and a manually-obtained refresh token — before any real API call could be made at all. That setup (below) is now done for TitanZ; a new business being added later just needs the same manual steps repeated for its own account, not more code.

**What Stage 0 required, and what was actually true once real credentials existed:**

1. **A Developer Token can only be issued to a Manager (MCC) account, never a standalone one.** Confirmed directly against TitanZ's own account — its API Center required a Manager account before it would issue a Developer Token at all. The actual setup: one Manager account for this platform (a separate signup, not TitanZ's own account), TitanZ's existing standalone account linked to it as a **client**, and the Developer Token obtained from the *Manager's* API Center. Every real API call needs a `login-customer-id` header set to the Manager account's own Customer ID, alongside the target business's `customerId` in the URL — see Credential storage below (the global `loginCustomerId` field this added).
2. **A fresh Developer Token starts restricted to Test-accounts-only access**, and querying a real, live account needs **Basic access** approval (a Google review). TitanZ applied for and received this before real leads could be pulled.
3. **The OAuth consent screen needs the authorizing Google account added as a Test user** while unverified, or Google blocks the flow entirely ("Access blocked: has not completed the Google verification process"). Also worth knowing long-term: an unverified app's refresh tokens for this scope (`adwords`, one of Google's "sensitive" scopes) expire after 7 days — fine for the testing done here, but real production use will eventually need the OAuth consent screen submitted for verification so the refresh token stops expiring weekly.
4. **The real JSON shape** of `local_services_lead`/`local_services_lead_conversation` — see "Real bugs found against live data" below for the full list of what the docs/proto definitions didn't make obvious until a real query was actually run.
5. **Mutability** — not empirically re-tested via a deliberate re-poll-after-a-few-minutes check, but the upsert design below was already built defensively regardless of the answer, so this was never a blocker either way.

## Credential storage — a third shape

Two credential shapes already exist in this codebase, and neither fits cleanly:
- **Twilio** is pure-global — one master account shared by every business, no per-business identity at all (see [call-dashboard.md](call-dashboard.md#one-master-twilio-account-not-one-per-business), a design mistake corrected once already this project).
- **ServiceTitan** is pure-per-business — every credential, including Client ID/Secret, is separate per tenant.

Google's OAuth model is neither: one registered OAuth "app identity" (Client ID/Secret, plus the Developer Token that identifies this platform's use of the Google Ads API as a whole) can mint access tokens against many separate end-users' own Google Ads accounts. So the split here is:

- **Global** (`settings` table, `googleAds.developerToken`/`clientId`/`clientSecret`/`loginCustomerId`) — this platform's own registered app identity, plus (confirmed necessary, see unknown #1 above) the Manager account everything runs through: `loginCustomerId` is the Manager (MCC) account's own Customer ID, sent as Google's required `login-customer-id` header on every real API call. One Manager account serves every business on this platform — a Manager account is designed to have many client accounts linked under it, so this stays a single global value even as more businesses' Google Ads accounts get linked as clients. Configured once on the platform-wide Admin Settings page (`getGoogleAdsPlatformConfig()`/`getRawGoogleAdsSettings()` in `src/settings/store.ts`, `GET`/`PUT /google-ads-settings` in `src/api/adminRouter.ts`), same encrypted-`settings`-table pattern as Twilio's/SMTP's credentials (`loginCustomerId` itself isn't a secret — an account number, not a token — so it's stored and displayed in plain text, not password-masked, same treatment as a business's own `customerId` below).
- **Per-business** (`business_settings` table, `googleAds.customerId`/`refreshToken`) — each business's own separate Google Ads account (its 10-digit Customer ID — once linked as a **client** under the Manager account above) and its own refresh token authorizing this platform's app identity to act against that account. Configured on that business's own General Settings page, same "leave blank to keep" pattern as its ElevenLabs API key.

Both are combined behind one strict getter, `getGoogleLsaConfig(businessId)` in `src/settings/store.ts` — callers never need to know or care which table a given field actually lives in, matching how every other integration's config getter already works. Keyed `googleAds.*`, not `googleLsa.*`, in both tables — these are Google-Ads-account-level credentials, not LSA-product-specific, so a future non-LSA Google Ads integration for the same business could reuse the same stored refresh token/customer ID without a new settings section.

**No in-app OAuth "Connect with Google" flow.** Matching how ServiceTitan/ElevenLabs credentials are handled everywhere else in this app, the refresh token is manually obtained (via Google's OAuth 2.0 Playground, during Stage 0) and pasted into the settings UI — not a self-service consent-screen flow. That could be built later if this expands beyond one or two businesses, but isn't needed to get TitanZ working first.

## `inbound_leads` upsert semantics for a polling source

Every lead source built before this one (the website form/chat webhook) is a one-shot submission — a given `external_id` is either new or a harmless exact redelivery, so the original `insertInboundLead()` used `ON CONFLICT ... DO NOTHING`. A polling source is different: the *same* `local_services_lead` resource can legitimately be seen again on a later poll with genuinely new content (a message thread's later reply, a call's billing data settling) — silently ignoring it on conflict would mean losing real content, not just no-op'ing a redundant delivery.

`insertInboundLead()` was upgraded to `ON CONFLICT(business_id, source, external_id) WHERE external_id IS NOT NULL DO UPDATE SET` on the content columns only — `name`, `phone`, `email`, `message`, `raw_payload_json` — explicitly **excluding** the staff-set triage columns (`status`, `is_read`, `internal_notes`), mirroring `upsertCallTranscription()`'s exact reasoning ([call-dashboard.md](call-dashboard.md#new-staff-set-columns--readunread-recoveredNot-recovered-the-bookabilitycall-reason-overrides-and-internal-notes)): a redelivered/re-polled row should refresh its own content, but must never silently reset something a human already triaged. This is a safe, backward-compatible change regardless of what Stage 0 finds — no source that's actually live today ever supplies `external_id`, so nothing currently shipped changes behavior; it only starts mattering once a polling source (this one) exists.

## Built — `src/googleLsa/`

- `authClient.ts` — refresh-token → access-token exchange, cached per business, mirroring `servicetitan/authClient.ts`'s expiry-minus-60s cache.
- `httpClient.ts` — a thin GAQL search wrapper via plain `axios` (`POST https://googleads.googleapis.com/v{version}/customers/{customerId}/googleAds:search`), not the official `google-ads-api`/`google-ads-node` SDK (gRPC/protobuf, opinionated config) — consistent with how every other integration in this codebase (ElevenLabs/Twilio/ServiceTitan) is plain REST via axios. Also normalizes `customerId`/`loginCustomerId` by stripping non-digit characters before use (see Real bugs below — Google's own UI shows these dashed, the API rejects dashes).
- `leads.ts` — field-mapping from the real, confirmed `local_services_lead`/`local_services_lead_conversation` shape into `insertInboundLead()`'s input shape, `source: "google_lsa"`, `externalId` = the lead's own resource name. `MESSAGE` leads join every matched conversation's text chronologically; `PHONE_CALL` leads get a duration + recording-URL summary; anything unmatched falls back to a plain key/value dump — same "never silently lose data" precedent as the generic webhook's `leadIntake.ts`.
- `pollLeads.ts` — mirrors `src/twilio/pollCalls.ts`'s `isPolling` boolean guard, loops `listBusinesses()` (LSA accounts are genuinely separate per business, unlike Twilio's one shared account matched by phone number), wired into `src/index.ts` on a 5-minute interval.

### Real bugs found and fixed against live data (2026-07-17)

None of these were guessable from documentation alone — each was found by actually running the poller against TitanZ's real, populated account and reading the real error/response:

- **API version drift**: the first-draft code guessed `v18`; by the time real credentials existed, `v18` had been fully sunset (a plain `404`, not a deprecation warning) and the current version was `v24`. Google Ads API versions are removed from routing entirely once retired — there's no soft-fail. Bump `API_VERSION` in `httpClient.ts` again once Google moves past `v24`.
- **Customer IDs must be digits-only in the URL/header.** Google's own UI (and this app's settings-page placeholder, `"123-456-7890"`) shows account IDs with dashes — the API 404s if you send them that way in the `customers/{id}` URL segment or the `login-customer-id` header. Fixed by stripping non-digits in `httpClient.ts` rather than asking anyone to type it a specific way.
- **GAQL's selectable-field rules are inconsistent across resources, not a fixed pattern.** `local_services_lead.contact_details` is selectable as a whole compound field — no error. But `local_services_lead_conversation.message_details`/`phone_call_details` are **not** selectable whole (`PROHIBITED_FIELD_IN_SELECT_CLAUSE`) and must be broken into their individual leaf sub-fields (`message_details.text`, `phone_call_details.call_duration_millis`, `phone_call_details.call_recording_url` — confirmed against the real `.proto` definitions). Trying to leaf-explode a field that's actually fine selected whole (like `contact_details`) gets a *different* error (`UNRECOGNIZED_FIELD`) — there's no way to know which rule applies to a given resource without just trying it against a real account.
- **Conversations query ordering had to match the leads query's recency window.** `LEADS_QUERY` fetches the 50 *newest* leads (`ORDER BY creation_date_time DESC`). The conversations query originally used `ORDER BY event_date_time ASC LIMIT 200` — which, for an account with any real history, fetches the *oldest* 200 conversations instead, never overlapping with those 50 recent leads at all. Confirmed directly: a real `MESSAGE` lead came back with an empty `conversations` array and fell through to the generic fallback message, purely from this ordering mismatch, not because the conversation didn't exist. Fixed by querying `DESC` (same recency direction as the leads query) and re-sorting each lead's own conversation group back to ascending afterward, so a multi-message thread still reads oldest-first.
- **A selected-but-inapplicable nested field comes back as an empty `{}` object, not absent.** A `PHONE_CALL`-channel conversation row still has a `messageDetails` key in the response (just `{}`), and a `MESSAGE`-channel row still has `phoneCallDetails` (also `{}`). Since `{}` is truthy in JS, `conversations.find((c) => c.phoneCallDetails)` would have matched the wrong row type. Fixed by checking the actual leaf value (`c.phoneCallDetails?.callDurationMillis`) instead of the parent object's mere presence.

### What's still unconfirmed

- The exact enum string values for `leadType`/`leadStatus`/`conversationChannel`/`participantType` beyond what's actually been seen (`MESSAGE`, `PHONE_CALL`, `ACTIVE`, `CONSUMER`) — if a lead ever shows up with a different value, `buildMessage()`'s type checks are the first thing to check.
- Mutability of a `local_services_lead` resource across repeated polls (does an ongoing thread's later reply show up as a content change on the same lead, or as new conversation rows only) — not deliberately re-tested via a poll-then-wait-then-poll-again check. The upsert design in `inbound_leads` (below) was already built defensively regardless of the answer, so this was never a blocker.
